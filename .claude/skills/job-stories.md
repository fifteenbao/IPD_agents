# Job Stories（场景化用户故事）

## 与传统 User Story 的区别

**传统 User Story**：
> 作为 [用户角色]，我想要 [功能]，以便 [收益]。
> "作为跑者，我想要查看配速，以便了解训练进度。"

问题：**抹掉了情境**——什么时候、为什么、什么阻力？

**Job Story**：
> When {情境}，I want to {动机}，so I can {预期结果}。
> "**当我跑到第 8 公里腿开始发酸时**，我想要设备主动给我配速反馈，**这样我就不用低头看手腕**，可以保持节奏到终点。"

智能硬件的核心价值往往**藏在情境里**：什么环境光、什么动作姿势、什么情绪状态、什么信号触发。Job Story 把这些显性化。

---

## 核心三段式

| 段落 | 关键问题 |
|------|---------|
| **When (情境)** | 什么时间、地点、活动、情绪、生理状态、设备状态？ |
| **I want to (动机)** | 用户想要的是结果还是手段？拆到不能再拆 |
| **so I can (结果)** | 真正的目标是什么？通常比动机深一层 |

---

## 撰写规则

1. **情境必须具体**——"跑步时" → "跑到 8 公里后腿开始发酸时"
2. **动机要中性**——不要预设方案。"我想要语音播报" 是方案；"我想要不看屏幕也能获得信息" 是动机
3. **结果要回到价值**——"看到配速" 不是结果，"保持节奏到终点" 才是
4. **每条独立**——不要把多个故事压在一句话里
5. **覆盖完整生命周期**——不只是核心使用，还要涵盖开箱、配对、充电、异常、维护、退役

## 智能硬件常见场景模板

### 开箱与首次使用
```
When 我刚打开包装并取出设备，
I want to 在不阅读说明书的前提下完成配对与初始化，
so I can 在 5 分钟内开始第一次使用。
```

### 户外极端环境
```
When 我在零下 5 度的山顶戴着手套，
I want to 用一次按压（不需要解锁、不需要看屏）触发记录，
so I can 不脱手套也能保留这段轨迹。
```

### 异常与恢复
```
When 设备在使用中突然断连/没电/闪退，
I want to 重新通电后自动恢复到上一次状态，
so I can 不丢失正在进行的数据。
```

### 长期使用
```
When 我已经使用产品 6 个月，
I want to 看到自己的进步轨迹和体能变化趋势，
so I can 获得持续的成就感（而不是淡忘）。
```

### OTA 与维护
```
When 半夜 3 点设备自动推送固件升级，
I want to 升级失败时设备能自动回滚到稳定版本，
so I can 早晨醒来设备依然能正常使用。
```


---

## 与其他 skill 的衔接

- **OST 的 Opportunities** → 转化为 Job Stories 的 When + I want
- **KANO 模型** → 必备需求 = Must 级 Job Story；魅力需求 = Should/Could 级 Job Story
- **PRD 模板的"用户故事"章节** → 直接填入 Job Stories
- **PRD 模板的"验收标准"章节** → 每条 Job Story 的 "so I can" 是验收的检验依据
