# Assumption Prioritization（假设优先级排序）

## 为什么硬件项目要做假设管理

软件假设错了：改代码、推送、修复。
硬件假设错了：模具报废、PCB 改版、认证重测、库存清仓。

**假设是项目的隐性炸药**——它们看起来像"事实"，但其实从未被验证。本 skill 把它们显性化、排序化、验证化。

---

## 假设的六大类别

| 类别 | 典型假设 |
|------|---------|
| **用户假设** | 目标用户愿意为 X 功能付 Y 元；用户能接受 Z 操作流程 |
| **价值假设** | 这个产品比竞品的差异化能转化为购买决策 |
| **可行性假设** | 主控芯片能跑下我们的算法；天线性能能过 FCC |
| **可用性假设** | 用户首次开箱能 5 分钟内完成配对；跌落 1.5 米不损坏 |
| **商业假设** | BOM 量产能压到 ${X}；渠道毛利留 30% 仍有竞争力 |
| **供应链假设** | 关键器件 12 周内能交付；首批 5000 台模具能撑过 |

## 三步流程

### Step 1 — 识别（不超过 30 min）
逐章扫描 Charter / PRD / 财务报告 / 技术报告，标出所有"假设性陈述"。判断标准：

> 把这句话改成相反描述，项目还能成立吗？如果不能，它就是一个假设而不是事实。

例：
- "BOM 成本 ¥180" → 假设（量产价格）
- "FCC 认证 8 周" → 假设（可能 16 周）
- "已签订采购合同 1 万台" → 事实（有合同）

目标：列出 30–50 条假设。

### Step 2 — 矩阵评分

```
                    影响（Impact）
                  低         中         高
            ┌─────────┬─────────┬─────────┐
        高  │ 低优先级 │ 中优先级 │ ★ 必验证 │
            ├─────────┼─────────┼─────────┤
不确  中    │ 忽略    │ 低优先级 │ 中优先级 │
定性        ├─────────┼─────────┼─────────┤
        低  │ 忽略    │ 忽略    │ 监控    │
            └─────────┴─────────┴─────────┘
```

- **影响**：若假设错了，对项目的损害（财务/时间/客户）
- **不确定性**：当前我们对它的把握（基于数据 vs 基于猜测）

### Step 3 — Top 5 必验证 + 验证手段

| 假设 | 影响 | 不确定性 | 验证手段 | 截止 | 负责 |
|------|------|----------|---------|------|------|
| 用户愿付 ¥499 | 高 | 高 | 价格梯度调研（n≥30） | {日期} | CMO |
| 主控能跑算法 | 高 | 高 | 在 EVT 板烧录 demo 测试 | {日期} | CTO |
| 模具厂 6 周交付 | 高 | 中 | 现场考察 + 提前下定 | {日期} | PM |

**验证手段优先级**（成本从低到高）：
1. 二手数据（行业报告、公开数据）
2. 用户访谈（10–15 人）
3. 纸面原型 / 假落地页
4. 工程样机（POC 板）
5. 小批量打样

## 输出格式

```
【假设清单 · {项目名}】
日期：{YYYY-MM-DD} | 阶段：{PDCP / TR3 / EVT}

━━━ 假设全集 ━━━
| # | 类别 | 假设描述 | 影响 | 不确定性 | 象限 |
|---|------|---------|------|----------|------|
| 1 | 用户 | 用户愿付 ¥499 | 高 | 高 | ★必验证 |
| 2 | 技术 | 主控能跑 NPU 推理 | 高 | 高 | ★必验证 |
| ...

━━━ 必验证 Top 5 ━━━
1. {假设} → 验证 {手段} — 截止 {日期} — 负责 {Agent}
2. ...

━━━ 验证完成后行动 ━━━
- 假设成立 → 进入下阶段
- 假设不成立 → 触发方案调整或 DCP 重审
```
